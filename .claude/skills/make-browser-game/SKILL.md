# ブラウザゲーム制作フロー

このサイト(nrtkumi.github.io)に1人用ブラウザゲームを追加するときの手順。
過去の制作(games/ 配下)で確立した流れと教訓をまとめている。

## 0. 入り方の判定と要求仕様

入り方は2つある。まずどちらかを判定する:

- **アイデアなし**(「何か新しいゲームを作って」): 1の市場調査から始める
- **コンセプト・要望あり**: 市場調査はスキップしてよい(題材の事実確認やAPIの実在確認が必要なときだけ調査する)。要望を以下に整理してから2へ:
  - **必須要件**(ユーザーが明示したこと)と**裁量部分**(こちらで決めてよいこと)を分ける
  - 裁量部分は2の設計観点チェックリストで自分で決め、**決定と理由を設計メモに残す**。逐一質問に戻らない
  - ユーザーに確認するのは、要望同士が矛盾する・技術的に実現不能(必要なAPIが存在しない等)のときだけ

## 1. 市場調査(アイデアがないときのみ)

**`/deep-research` スキルを使って調査する**(WebSearchを直接叩かない)。複数ソースをファンアウト検索し、出典付きでファクトチェックされたレポートが得られるので、企画の根拠とREADMEの「作成の背景」にそのまま使える。

調査の観点(これらを束ねた具体的なリサーチ質問を deep-research に渡す):

- 直近のヒットゲーム・バイラルゲームのメカニクス(国内外)
- ハイパーカジュアル/カジュアルゲームのトレンド
- インディーゲームのトレンド(ローグライト要素など)

リサーチ質問の例(年は当年に置き換える):
> 「<年>に流行しているブラウザ/モバイル向け1人用ゲームのメカニクスとトレンドを調べたい。バイラルになったハイパーカジュアル/カジュアルゲーム、インディーのローグライト要素、国内外のヒット作のコアメカニクスを、具体例と出典付きで挙げてほしい。スマホのタッチ操作のみ・単一HTMLで実装できる規模のゲームに応用する前提。」

注意:

- 質問が漠然としすぎている場合は deep-research が2〜3個の絞り込み質問を返すので、上記の前提(スマホファースト・単一HTML・1人用)を織り込んで答える
- 得られたレポートの要点と出典URLを記録する(後でREADMEの「作成の背景」に載せる)

## 2. 企画・設計

### 基本条件(1案に絞る)

- **1文で説明できるコアメカニクス**(5秒で理解できる操作)
- スマホファースト(タッチ操作のみで完結、縦画面)
- 単一HTMLファイルで実装できる規模
- 市場調査をした場合はトレンド要素を1つ以上入れる(例: ローグライト強化、マージ、成長システム)
- 既存ゲームとの被りは**ゲーム名でなく実メカニクスで判定する**(games/*/README のコアループを確認。名前が違っても中身が同型のことがある)。被らないと望ましいが必須ではない — 面白さと実装の質を優先し、被りを理由に良い案を却下しない

### 設計観点チェックリスト(全観点に答えてから実装に入る)

コンセプトと要望だけが与えられた場合も、以下を自分で埋めて設計を完成させる。要望と矛盾しない限り自分の判断で決め、理由ごと設計メモに残す。

**A. コアループとセッション**
- 操作を1文で書けるか。操作の種類は1〜2つに絞れているか
- 1プレイの想定時間(目安30秒〜3分)。1プレイ内のテンポ — 手番の間の死に時間・結果待ちの無反応時間がないか
- 「もう1回」の動機は何か: スコア更新/ランダムな引きの偏り/上達の実感、のどれで引っ張るか

**B. 難易度と最適戦略(バランス破綻の事前検討)**
- 序盤(最初の30秒)は必ず成功体験になるか。初見が理不尽に死なないか
- **単調な最適戦略が存在しないか机上で検討する**: 「常に安全択」「常に強気」が最適なら設計が死んでいる。実例: 加算報酬のプッシュユアラックは即降りが最適になる → 乗算報酬+資源/時間の上限で内点最適(押し引きの分岐点)を作った(games/bakuage)
- 実力とランダム性の比率: 上級者がハイスコアを更新し続けられる余地があるか
- 数値(出現率・係数・閾値)は仮置きでよいが、**後でボット調整する前提でコード先頭の定数に括り出す**設計にしておく(4参照)

**C. スコア設計**
- 初回プレイの期待スコア/上級者の到達点/理論上の天井をラフに見積もる
- リスクリターン(コンボ、倍率、賭け)を1つは入れる
- ベストスコアはlocalStorageに保存。通算値・プレイ回数もあると再訪動機になる

**D. フィードバック(ジュース)**
- 全ての操作に即時の視覚+聴覚フィードバックを返す
- 成功に段階を付ける(通常<良い<最高)。パーティクル・画面シェイク・数字ポップ・コンボ演出
- 状態の区別を色だけに依存しない(形・位置・ラベルを併用)

**E. オンボーディング**
- チュートリアル画面を作らずに済む設計を最優先(操作1種類、画面が次の行動を誘導する)
- スタート画面: 1行ルール+操作説明+ベストスコア表示
- ルール説明が3行を超えるなら複雑すぎるサイン → 企画に戻って削る

**F. 操作エルゴノミクス(スマホ縦持ち)**
- 重要な操作は画面下半分(親指リーチ内)。上部は表示専用にする
- タップターゲットは44px以上、隣接要素と間隔を空ける
- 連打・誤タップ対策: ゲームオーバー直後のリトライボタンは0.5秒ほど無効化(プレイ中の連打がそのままリトライを押す事故を防ぐ)
- 中断耐性: visibilitychange/blurで自動ポーズし、経過時間で不利にならないようにする

**G. ゲームオーバーとリトライ**
- 結果画面に内訳(最大コンボ、到達ステージ等)を出し、「次はこうしよう」が浮かぶようにする
- リトライは1タップで即時(スタート画面に戻さない)
- 理不尽死の排除: 反応不可能な速度、予告なしの即死、画面外からの死をなくす

**H. 題材・データ(外部API・実データを使う場合)**
- **鍵不要・CORS可を実証してから企画を確定する**。curlとブラウザ両方で確認 — ヘッドレスやcurlで通っても実機iOS WebKitで死ぬAPIがある(実例: iTunes media=music)
- レート制限と取得失敗時のフォールバック(キャッシュ、同梱データ、リトライ)を設計に含める
- 情報の出し方でネタバレしないか(実例: YouTubeはタイトルに答えが書いてある → サムネだけ見せて正解後に再生)
- 実証済みAPIと罠は既存ゲームのREADME(sateist、bakuage、pitari 等)とメモリに記録がある。まずそこを確認する

**I. スコープ**
- 1セッションで実装→テストまで到達できる粒度に切る
- 削ってよいもの: 図鑑、実績、複数モード。**削ってはいけないもの: ジュース、効果音、リトライ導線、ベストスコア保存**

### デザインコンセプト(必須)

**見た目デザインは既存ゲームから踏襲しない**。ユーザーから指定がない場合は、実装前にそのゲーム専用のデザインコンセプトを都度考えて適用する:

- 題材・世界観から出発して、配色パレット・タイポグラフィ・質感・モチーフを言語化してから作る(例: 「昭和の銭湯ポスター」「90年代アーケード筐体」「活版印刷の番付表」)
- **AI感の出る定番を避ける**: 紫系グラデーション、グラスモーフィズム、ネオングロー一辺倒、意味なく丸いカード+影、どのゲームでも同じフォント、といった「生成AIが作りがちな見た目」をデフォルトにしない
- フォントもコンセプトに合わせて毎回選ぶ(Google Fontsの日本語フォントは選択肢が多い)。既存ページとの統一は不要
- 決めたコンセプトはREADMEに一言で記録する

### 設計メモと設計レビュー

- A〜Iの回答+デザインコンセプト+画像アセット計画を短い**設計メモ**にまとめる(scratchpadでよい。READMEと最終報告の素材になる)
- 実装前に **codex に設計メモをレビューさせる**(「最適戦略が単調にならないか」「破綻する入力・状態はないか」「もっと面白くする一手はないか」を突かせる)。実装後のコードレビューと合わせて設計→コードの2段レビューが定着している
- まとまったら実装に進んでよい(ユーザーの確認待ちにしない。必須要件と矛盾する変更をしたいときだけ確認する)

## 3. 実装

`games/<ゲーム名>/index.html` に単一HTMLで実装する。

### ライブラリの活用(推奨)

体験や表現をリッチにするためなら**ライブラリを積極的に入れて良い**(例: three.js での3D表現、物理エンジン、トゥイーン/パーティクル系)。実例: MEZAME は three.js で3D描画している。

- 読み込みはCDN(`<script src>` や import map + ES modules)で行い、単一HTML構成は維持する
- バージョンはURLに固定する(`@0.x.y` 形式)。CDN読み込み失敗時はエラーメッセージを出すなど無言で壊れないようにする
- 「ライブラリで表現がどれだけ化けるか」を企画段階で一度検討する。素のcanvasで十分ならそれでも良いが、デフォルトで避ける理由はない

### 必須チェックリスト(スマホ対応)

- `<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover, user-scalable=no">`
- セーフエリア: `env(safe-area-inset-*)` でホームバー回避
- `touch-action: none`(canvas)、`-webkit-tap-highlight-color: transparent`、`user-select: none`、`overscroll-behavior: none`
- **ダブルタップズーム無効化** — iOS Safariは`user-scalable=no`を無視するので連打でズームしてしまう(tanetoriで実際に起きた)。3点セットで防ぐ:
  1. 全要素に `touch-action: manipulation`(canvas/ボタンは`none`のまま)
  2. JSで `dblclick`/`gesturestart` をpreventDefault + 350ms以内の連続`touchend`をpreventDefault
  3. touchendのpreventDefaultは合成clickを潰すため、**`click`イベントに依存しない**(Pointer Eventsで拾う)
- 入力はPointer Eventsで統一(マウス兼用)。上記の通り`click`は使わない
- **flexレイアウト列の中のcanvasは `flex: 1 1 0; min-height: 0`** — `auto`だと実機iOSで下に溢れる。ヘッドレスでは再現しないので気づきにくい(実際に起きた)
- **PCワイド画面対策** — ゲーム内サイズを横幅から比例計算していると全幅表示で巨大化する(tanetoriで実際に起きた)。ルート要素を `max-width: min(520px, 64vh)` 程度の縦長カラムに制限して中央寄せ(`left: 50%; transform: translateX(-50%)` + 暗背景・シャドウ)。キーボード操作(←→/A・D、Space等)も付ける
- フォントはデザインコンセプトに合わせてゲームごとに選ぶ(上の「デザインコンセプト」参照)

### 必須チェックリスト(iOS音声) — 過去のバグの再発防止

Web Audioの音はiOSの消音スイッチ(マナーモード)で無音になる。必ず以下を入れる:

1. 初回タップ時に**無音WAVを `<audio>` 要素でループ再生**(オーディオセッションを再生カテゴリへ切り替える。無音WAVはJSでArrayBuffer生成→Blob URL)
2. `AudioContext.resume()` の**Promise完了を待ってから**アンロック判定
3. `AC.onstatechange` で `interrupted` からの自動復帰
4. 実装例は `games/hakai/index.html` の audio セクションを参照

### 必須チェックリスト(設計)

- **メインループ(rAF)はスタート画面の時点から回る。ループが触る状態は全て宣言時に初期化する**(init()任せにすると初回フレームの例外でループが死ぬ — hakaiで実際に起きたバグ)
- **バランスに関わる数値はコード先頭の定数ブロックにまとめる**(4のボット調整で触るのはここだけにする)
- ゲームロジックはDOM非依存の純粋関数に寄せる(ヘッドレステストとボットシミュレーションでそのまま回せる)
- 効果音はWeb Audioでリアルタイム合成(外部音源不使用)
- スコア等の永続化は localStorage(try/catchでフォールバック)
- 長いセッション向けに速度ボタンや自動加速を検討

### 必須チェックリスト(OGP・メタタグ)

リリース前ではなく**実装の段階でメタタグを入れておく**(後回しにすると忘れる):

- `<meta name="description">` と OGP一式: `og:title` / `og:description` / `og:image` / `og:type`(website)/ `og:url` + `twitter:card`(summary_large_image)
- `og:image` と `og:url` は**絶対URL**(https://nrtkumi.github.io/...)で書く
- 既存例: `games/tanetori/index.html`、`games/suika-fever/index.html`
- 画像は **1200x630**(またはタネトリの1200x800)、JPEG可。置き場所は `games/<ゲーム名>/assets/ogp.jpg`
- 画像の作り方は2通り。ゲームの見た目が伝わる方を選ぶ:
  1. **ゲームプレイのスクリーンショット**(進行中の画面が一番伝わる) — Playwrightで自動プレイさせて撮る。`deviceScaleFactor: 2` + 600x315のclipでちょうど1200x630になる。盤面が育つまで数十秒プレイさせ、複数タイミングで撮って目視で選ぶ。ポインタを左右にスイープすると操作が散って画になる(実例: スイカフィーバーのOGP制作)
  2. **画像生成** — codex経由のgpt-image-2でタイトルロゴ入りのキービジュアルを生成する(下の「画像アセット」参照)。プレイ画面がOGP映えしないゲームや、文字主体のビジュアルにしたい場合はこちら
- スクショ撮影の注意: 長時間の自動プレイでcanvasの2Dコンテキスト状態がリセットされ、初期化時の `ctx.scale(dpr, dpr)` が消えて描画が崩れることがある(スイカフィーバーで実際に起きた)。**dpr変換は毎フレーム `ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` で再設定**しておく(iOS Safariのコンテキスト喪失対策にもなる)

### 画像アセット(推奨) — codex経由の画像生成でリッチにする

**見た目をリッチにする画像は積極的に生成して使う**。絵文字+canvas描画だけで済ませず、実装の段階で「ここに画像があれば化ける」箇所を洗い出し、最低1〜2枚は入れることを検討する(例: タイトル画面のキービジュアル、ゲーム背景、キャラ・札・アイテムのスプライト、図鑑用イラスト)。**Codex CLI(`codex`)経由でgpt-image-2の画像生成**が使える:

- 使い方: シェルから `codex` を呼び出し「○○の画像を生成して games/<ゲーム名>/assets/xxx.png に保存して」のように依頼($imagegenスキル)
- ChatGPTサブスクリプション内で動作(API追加課金なし)。日本語テキストの描画にも対応
- スタイル指定(画風・背景色・構図)を固定の前置きにして対象だけ変えると、複数枚でも統一感が出る(実例: オーラキャッチのキャラ24枚バッチ生成)
- 生成は1024px PNG → `sips` で512px程度のJPEGに変換して軽量化(24枚で約1.4MBの実績)。ロード枚数が多い場合は遅延ロードや読み込み失敗時のフォールバック(絵文字/canvas描画)を必ず用意する
- 注意: gpt-image-2は**透過背景に非対応**。ゲーム背景と同色の単色背景で生成してcanvas側でクリップして馴染ませるか、白背景生成→flood fillで透過化する。gpt-image-1.5(API課金)なら透過対応
- READMEの技術構成に生成手段とプロンプトの要点を記録する

## 4. バランス調整(実装後・テスト前に必ずやる)

数値調整は感覚でなく**ボットシミュレーション**で行う:

- ゲームロジック(3で純粋関数に寄せた部分)をNodeで回し、戦略の違うボット(ランダム/常に安全/常に強気/最適風)で**各1万ラン**程度
- 見るもの:
  - スコア分布 — 初心者想定と上級者想定の差が付いているか、天井に張り付いていないか
  - 平均プレイ時間が設計メモのセッション想定と合っているか
  - **単調戦略のボットが勝ちすぎていないか**(勝っていたら2-Bの設計に戻って報酬構造を直す)
- **完全情報ボットは実プレイヤーより1〜2割甘く出る**(見落とし・反応遅れがない)。ボット基準で「ちょうど良い」は実機では少し辛いので、易しめに寄せる
- 調整はコード先頭の定数ブロックだけで完結させ、調整結果(ラン数と主要指標)をREADME用に記録する
- 仕上げに codex にコードレビューをさせる(設計レビューと合わせて2段レビュー)

## 5. テスト(リリース前に必ず全部やる)

1. **構文チェック**: インラインscriptを抽出して `node --check`
2. **ヘッドレスロジックテスト**: DOMをスタブしてNodeでゲームループを回し、開始→操作→進行→ゲームオーバー→記録保存を通す
3. **実ブラウザE2E(最重要)**: puppeteer-core + ローカルChromeで、モバイルビューポート(`hasTouch: true`)+タッチイベントで実際にプレイする
   - `pageerror` / consoleエラーがゼロであること
   - canvasの描画ピクセルがあること(getImageDataで確認)
   - 数ターン操作してスコア・進行が動くこと
   - スクリーンショットを撮って見た目を目視確認
   - 過去、ヘッドレステストだけでは「スタート前のフレームで例外→ループ停止」を見逃した。**実ブラウザE2Eを省略しない**
   - 雛形: `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome` を executablePath に、一時ディレクトリに `npm install puppeteer-core`
   - **`file://` ではなくローカルHTTPサーバ経由でテストする**(`python3 -m http.server`)。file://だと同梱画像を読んだ時点でcanvasが汚染され `getImageData` がSecurityErrorになる(SOZO 8192で実際に起きた)

## 6. ドキュメント

`games/<ゲーム名>/README.md` を作成:

- 概要・プレイURL・対応環境
- 遊び方・ルール(強化があれば一覧表)
- 作成の背景(市場調査の要点と出典リンク。コンセプト持ち込みの場合は設計メモの要点 — 何を狙ってどう設計したか)
- バランス調整の記録(ボットのラン数と主要指標、調整の意図)
- 技術構成(単一HTML、iOS音声対策、使用API・生成画像など)
- 実施したテスト内容と、開発で得た教訓

## 7. 公開

1. `git add` → コミット(本文に変更概要、Co-Authored-By付き)→ `git push`(mainがそのままGitHub Pages)
2. ファイルを移動した場合は旧URLに meta refresh + `location.replace` のリダイレクトHTMLを残す
3. ユーザーにプレイURL(https://nrtkumi.github.io/games/<ゲーム名>/)を案内。反映に1〜2分かかる旨を添える
