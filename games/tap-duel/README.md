# TAP DUEL(タップデュエル)

スマホ1台を2人で挟んで、画面の上下半分ずつを使って対戦する5点先取のミニゲーム対決。

- **プレイURL**: https://nrtkumi.github.io/games/tap-duel/
- **対応環境**: スマホ(縦画面・タッチ操作)ファースト。マルチタッチ対応。PCのマウスでも動作(交互操作)。依存ライブラリなしの単一HTML。

## 遊び方

1. スマホをテーブルに置き、2人で向かい合って挟む(下半分が青のP1、上半分がピンクのP2。上半分は180°回転表示)
2. どちらかが画面をタップすると対戦開始
3. 毎ラウンド、18種のミニゲームからランダムに1つ出題(同じゲームは連続しない)
4. ラウンドに勝つと1ポイント。**先に5ポイント取ったほうが勝利**
5. 通算勝敗は端末の localStorage に保存される

## 作成の背景

「スマホ1台のローカル対戦ミニゲーム集」は定番ヒットジャンルで、JindoBlu「[2 Player Games: the Challenge](https://play.google.com/store/apps/details?id=com.JindoBlu.TwoPlayerGamesChallenge)」(短時間で勝負がつくミニゲーム集・オフライン完結)が代表例。ブラウザでも [12 Mini Battles 系の同端末対戦](https://twozygames.com/blog/best-2player-games/)や[同端末マルチプレイ](https://minireview.io/top-mobile-games/best-same-device-multiplayer-games-mobile)が人気を維持している([国内まとめ](https://app-liv.jp/games/casual/3499/))。サイト内の既存ゲーム(HAKAI=ブロック崩しローグライト、AURA CATCH=タイミングタップ収集、SUIKA FEVER=マージ)とジャンルが被らない「対面2人対戦」を採用した。

- コアメカニクスは1文で説明可能: 「スマホを挟んで、出題されるミニゲームで早押し対決、5点先取」
- 各ミニゲームのルールも1行で、出題時に両プレイヤーの向きで表示される

## 技術構成

- 依存ライブラリなしの単一 `index.html`(DOM + CSS、メインループは rAF)
- 上半分は `transform: rotate(180deg)` で対面プレイヤー向きに表示。中央バーは両向きにスコアを表示
- 入力は Pointer Events に統一(マルチタッチ対応。つなひきは2人同時連打が可能)。チキンレースの長押しは `pointerup` / `pointercancel` で離した瞬間を判定
- 効果音は Web Audio でリアルタイム合成。**iOS消音スイッチ対策**(初回タップで無音WAVを `<audio>` ループ再生 → `AudioContext.resume()` の完了待ち → `onstatechange` で復帰)は hakai と同方式
- メインループはタイトル画面の時点から稼働し、ループが触る状態は全て宣言時に初期化(hakaiの教訓)
- 通算勝敗は localStorage(try/catch フォールバック付き)
- フォントは既存ページと統一(Bungee + Zen Kaku Gothic New + Space Mono)

## 実施したテスト

1. **構文チェック**: インラインscriptを抽出して `node --check`
2. **ヘッドレスロジックテスト**: DOMスタブ+偽時計でゲームループを駆動し、タイトル→開始→全18種ミニゲームの勝敗判定→5点先取→勝利画面→localStorage保存→再戦、までを assert で検証(再戦をまたいで18種全てを網羅。まちがいさがし・ひかった順のおてつきロックも検証)
3. **実ブラウザE2E**: puppeteer-core + ローカルChrome、モバイルビューポート(390×844, `hasTouch: true`)+タッチイベントで実プレイ
   - `pageerror` / consoleエラー ゼロ
   - 全18種のミニゲームを実タッチで進行(マッチタップの正解タップ、数字タッチの順番タップ、つなひきの両ゾーン同時連打、ピッタリ5秒の両者ストップ、シグナル早押しのフライング判定、モグラ5連打、まちがいさがし、けいさん3択、チキンレース・ちょうど注ぎの長押し→解放、記憶3択、ストループの色一致判定、ゲージストップの両者停止、かぞえタッチの計数、ひかった順の再現、だまし早押しのフェイント回避、どっちが多い?の比較、両者のじゃんけん選択まで確認)
   - 上ゾーンの180°回転、スコアバーの更新をDOMで検証。スクリーンショットで見た目を目視確認

### 開発で得た教訓

- 対面2人プレイは「各プレイヤーのゾーンに同じHTMLを描き、上ゾーンごと回転」が最も単純で堅い。結果表示など視点依存の文言だけ `detail[player]` で出し分ける
- E2Eで時限イベント(強制マッチ出現など)を待つときは、ゲーム内の最悪ケース時間(本作は5.4秒)より長い監視枠を取る。固定sleepでなく結果テキストのポーリングで同期する
