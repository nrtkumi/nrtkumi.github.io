# ピタリ乗車 — 所要時間あてゲーム

2つの駅が発車標に表示されます。**最速で何分で着くか**を予想し、本物の乗り換え検索の結果とくらべて「ピタリ」を狙う、所要時間あての高得点ゲーム。

- **プレイ:** https://nrtkumi.github.io/games/pitari/
- **対応:** スマートフォン(タッチ)推奨 / PC(マウス)も可。縦画面・1人用・単一HTML。

---

## 遊び方

1. 上の発車標に **出発駅 → 到着駅** が表示されます。
2. スライダーと `±1 / ±10` ボタンで、**最速の乗車時間(分)**を予想して「決定」。
3. 本物の乗り換え検索の結果が開き、誤差の小ささで得点。実際の最速ルート(路線・乗換回数)も表示されます。
4. 全10区間の合計点で勝負。誤差が小さいほど高得点、連続ヒットで**コンボ倍率**が伸びます。

### 得点

| 誤差 | 判定 | 基礎点 |
|---|---|---|
| ±5%以内 | ピタリ！ | 1000 |
| ±10%以内 | ニアピン | 700 |
| ±20%以内 | おしい | 450 |
| ±35%以内 | まあまあ | 250 |
| ±55%以内 | ニアミス | 100 |
| それ以上 | 乗り過ごし | 0 |

「おしい」以上を連続で出すとコンボ。2連続目から ×1.5、×2.0 …と倍率が上がります。自己ベストは端末に保存されます。

## 作成の背景

**「Transit API(日本全国の乗り換え検索)で遊ぶ」**というお題に対し、価格当てバラエティ(The Price Is Right)の「ピタリ賞」を所要時間に置き換えた**推定当てゲーム**を採用しました。

- **なぜ面白い/中毒性:** 「だいたい何分かかるか」は誰でも直感を持てる一方、新幹線・快速・乗換の有無で実際は大きくブレる。**near-miss(あと少しでピタリ)**と自己ベスト更新欲が「もう1区間」を誘発します(ハイスコア更新ループ)。
- **データ源:** [Transit API](https://api.transit.ls8h.com) の `GET /api/v1/locations/suggest`(駅名→駅ID/座標の解決)と `GET /api/v1/plan`(2駅間の経路検索・最速所要時間)。読み取り専用・認証不要・CORS開放でブラウザから直接利用。
- **片手・縦画面設計:** スライダーと大きな数字、決定ボタンを親指の届く下半分に配置。

## 技術構成

- **単一HTML**(インラインCSS/JS、ビルド不要)。
- **デザインコンセプト:** 駅の**発車標(LEDドットマトリクス)**。黒地にアンバー/グリーンのLED、走査線、`DotGothic16` フォントで「方向幕」感を演出。AI感の出る紫グラデやグラスモーフィズムは不使用。
- 区間は出発駅・到着駅を毎回ランダムに選び、`plan` の最速ジャーニー(徒歩のみは除外、3〜300分の範囲)を正解値に採用。
- 効果音は Web Audio でリアルタイム合成。iOS の消音スイッチ対策(無音WAVループ + `AudioContext.resume()` 待ち + `interrupted` 復帰)を実装。
- スコアは localStorage に保存(try/catch フォールバック)。

## 実施したテスト

- インラインJSの構文チェック(`node --check`)。
- 実機相当の puppeteer E2E(iPhone ビューポート・タッチ操作、ローカルHTTPサーバ経由):複数区間をプレイし、`plan` の実データで正解時間が表示・採点され、`pageerror`/console エラーが出ない(favicon の404を除く)ことを確認。
- 開発で得た教訓:Transit API の駅ID(`feedId:stopId`)や日本語IDは**URLエンコード必須**。`plan` は徒歩のみのジャーニーも返すため、**transit leg を含む最速便だけを正解**に採用するフィルタが必要。
