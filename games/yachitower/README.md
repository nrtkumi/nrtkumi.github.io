# ヤチタワー（YACHI TOWER）

物件情報を読んで、**家賃が基準より「高い」か「安い」か**をタップで当てるハイ＆ロー。当てるごとにタワマンが1階ずつ伸び、ミスで崩壊するエンドレス・スコアアタック。

- **プレイURL**: https://nrtkumi.github.io/games/yachitower/
- **対応環境**: スマホ（縦持ち・タッチ）優先。PCはマウス＋キーボード（↑＝高い / ↓＝安い、Space＝開始）にも対応
- **デザインコンセプト**: 「不動産チラシ（マイソク）× タワマン」。紙質感の生成りベース、不動産レッド＋ネイビーの2色、明朝（Shippori Mincho B1）の見出しに Oswald の数字。手描き風の間取図を物件ごとに自動生成し、本物の賃貸チラシのような“スペックシート”として読ませる

## 遊び方

1. 画面上部の**基準物件**には家賃が表示されている（例: 中野区・1K / 23㎡ 徒歩7分 → ¥82,000）。
2. 下の**物件カード**には、間取図・専有面積・駅徒歩・築年数・所在階・構造・向き・設備が載っているが、**家賃は伏せられている**。
3. このカードの家賃が、基準より **高い** か **安い** かを推理して、`▲高い` / `▼安い` をタップ。
4. 正解すると家賃が公開され、**タワマンが+1階**。外したら**崩壊してゲームオーバー**。
5. 当てたカードが次の「基準物件」になり、新しい挑戦物件が出続ける。到達した**最高階**がスコア。

### 難易度

階が上がるほど、挑戦物件の家賃が基準に**僅差**で寄ってくる（floor0で平均±48%差 → floor20で平均±5%差）。直感だけでは届かなくなり、立地・広さ・築年・設備のトレードオフを総合判断する必要が出てくる。

### 5階ごとの強化（ローグライト）

5階に到達するたびに、3択から1つ選んで強化できる。

| 強化 | 効果 |
|------|------|
| 🛡 保険を確保 | ミスを1回だけ無効化（基準は更新して続行）。最大3つまでストック |
| ✨ 次は2階分 | 次に正解したとき+2階のぼる |
| 🔮 相場メーター | 以降5問、物件に『割安／適正／割高』タグが付き判断材料が増える |
| 🏗 基礎工事 | 即時+1階 |

### ランク（最高階で決定）

新人内見係 → 駆け出し仲介 → 一人前 → ベテラン仲介 → 大家マスター → **不動産王（60階〜）**

## 物件データについて

実在の物件APIは使っていない（CORS・鍵・在庫消滅の問題を避けるため）。代わりに、**実勢の賃貸相場を元にした生成モデル**で毎回ランダムな物件を作っている。家賃は次の要素の積で決まり、最後に個体差ノイズを乗せて1,000円（安価帯は500円）単位に丸める。

```
家賃 = 専有面積 × エリア㎡単価 × 面積係数(小さい部屋ほど㎡単価↑)
       × 駅徒歩係数 × 築年係数 × 所在階係数 × 構造係数 × 向き係数 × 設備補正 × ノイズ
```

- **エリア**: 東京23区＋川崎/横浜/武蔵野/さいたま/大阪/京都/神戸/名古屋/福岡/札幌など32エリア。港区・渋谷区・千代田区が高く、足立区・葛飾区・札幌市が安い、という実勢の序列を㎡単価で表現
- **間取り**: 1R〜3LDK。間取りごとに専有面積レンジと帖数配分を持つ。間取図SVGは部屋・水回り・玄関・バルコニーを手続き的に描画し、帖数は「室内有効面積」から逆算するので破綻しにくい
- 生成モデルなので、立地が良くても築古・遠い物件は安く、郊外でも新築・駅近は高い、といった“相場の逆転”が自然に起き、ゲームの判断材料になる

## 作成の背景

- 企画の出発点は賃貸物件検索サービス **comfy.maison**（[about](https://comfy.maison/about) / [開発記事](https://qiita.com/choo/items/2efacb2f94f6c9c46417)）。Yahoo!不動産から日次更新で**165万件超**の賃貸を取り込み、家賃・面積・間取り・駅徒歩・築年数・設備でサクサク絞り込める高速検索が売り。この「物件を構成する属性そのもの」をゲームの手がかりに転用した
- メカニクスは、視聴回数や数値の大小を二者択一で当てて連鎖させる **Higher Lower Game** 系の鉄板フォーマットを家賃に応用。5秒で理解でき、片手タップで無限に遊べてSNSでバズりやすい型を踏襲した
- 家賃という数字は「立地・広さ・築年・設備」の関数だが、人間の直感は曖昧——この“読めそうで読めない”レンジが、当て物としての面白さになっている

## 技術構成

- **単一HTMLファイル**（外部音源・画像なし。Google Fontsのみ）
- 描画: DOM（カード＝マイソク風スペックシート）＋ inline SVG（間取図を物件ごとに自動生成）＋ canvas（伸びる/崩れるタワマン、毎フレーム `setTransform(dpr,…)` でDPR再設定）
- 効果音: **Web Audio でリアルタイム合成**（正解チャイム／フロア上昇／崩壊ノイズ／強化）。iOS消音スイッチ対策として、初回タップで無音WAVを `<audio>` ループ再生し、`AudioContext.resume()` 完了とstatechange自動復帰を実装
- 永続化: 最高階を localStorage（try/catchフォールバック）
- スマホ対応: `viewport-fit=cover` ＋ セーフエリア、`touch-action`、ダブルタップズーム抑止（連続touchend/dblclick/gesturestartをpreventDefault、入力はPointer Events）、`max-width:min(460px,62vh)` の縦長カラムで中央寄せ＆PCワイド対策

## テスト

1. **構文チェック**: インラインscriptを抽出して `node --check`
2. **ヘッドレスロジックテスト**: DOM/AudioContext/localStorageをスタブし、家賃モデルを8,000件生成して分布検証（1Rの中央値¥65,500〜3LDK¥210,000で実勢妥当・NaNゼロ）。難易度ランプ（floor0:±48% → floor20:±5%）と、家賃同値（判定不能）ペアが0件であることを確認
3. **実ブラウザE2E（puppeteer-core＋ローカルChrome / `python3 -m http.server` 経由）**: モバイルビューポート（hasTouch）で開始→25ラウンド連続プレイ→強化選択→ゲームオーバーまでを通し、`pageerror`/console errorゼロ、canvas（タワー）の描画ピクセル存在を確認。各画面のスクリーンショットを目視

## 開発で得た教訓

- **canvasの属性サイズがflexレイアウトへ漏れる**: タワー用 `<canvas>` に幅・高さ属性（＝intrinsic aspect-ratio）を与えると、`flex:0 0 48px` 指定でもアスペクト比経由でCSS幅が1300px超に膨張し、隣のカードが4pxに潰れた。**CSSで `width`/`height` を両方明示**して属性サイズがレイアウトに干渉しないよう固定して解決
- **flex子孫の `min-height:0` 連鎖**: `#app` を `height:100svh` にしても、`.col`/`.card`/`.cbody` に `min-height:0` が無いと中身の最小コンテンツ高が上へ伝播して `#app` がビューポートを超え、スタート画面がスクロール必須になっていた（ヘッドレスでは数値計測しないと気づけない典型）
- **ハイ＆ローは“最大差を縮める”のが難易度**: 当初は最小差だけを保証していたが、ランダム物件は元々大きく散らばるため全く難しくならなかった。階に応じて**家賃差の上限**を縮めて接戦を作ることで、初心者でも入れて上級者には地獄、というカーブになった
