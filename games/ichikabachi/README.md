# 一か八か — 賭場のブラックジャック・ローグライト

引くか、勝負か。**21に近いほど配当が跳ねる**賭場のブラックジャック。軍資金を溶かさずノルマを越え、**縁起物(レリック)**で運を盛って、何場まで生き残れるか。実在トランプ画像で遊ぶ押し引きローグライト。

- **プレイ**: https://nrtkumi.github.io/games/ichikabachi/
- **対応環境**: スマホ縦画面ファースト(タッチ操作)。PCはキー操作対応(Space=引く / Enter=勝負 / 1=小・4=全)。
- **デザインコンセプト**: 昭和の地下賭場。黒漆に朱と金、麻の葉模様、筆文字(Yuji Syuku)。紫グラデやネオングローは使わない。

## 遊び方

1. 毎勝負、まず**賭け金**を選ぶ(小=10% / 中=25% / 大=50% / **一か八か=全額**)。
2. 自分に2枚、親に2枚(1枚は伏せ)が配られる。**引く**か**勝負(止める)**か。
3. 21を超えたら(バースト)即負け。止めたら親が17以上になるまで引く。
4. 勝てば配当。**21に近いほど配当倍率が高い**(21=×3 / 20=×2 / 19=×1.6 / 18=×1.3 / それ以下=×1)。押し引きの旨味。
5. 軍資金が**ノルマ**に届けば場をクリア。**縁起物**を一つ授かり、次の場はノルマが上がる。
6. 賭けられなくなったら(軍資金 < 10)破産。到達した場数と最高軍資金が記録される。

### 縁起物(レリック)

| 名前 | 効果 |
|---|---|
| 🗿 身代わり地蔵 | 1勝負に1度、バストを無かったことに(引いた札を捨てる) |
| 🎴 福の神 | ピッタリ21の配当が ×3 → ×5 |
| 👁️ いかさま札 | 親の伏せ札が最初から見える |
| 🌕 五光 | 5枚引いてバストしなければ自動的に勝ち(×3) |
| 🪙 金運 | 勝ったときの配当 +25% |
| 🛡️ 守り銭 | 負けても賭け金の半分が戻る |
| 🔥 連勝札 | 連勝するほど配当が伸びる(連勝数 ×0.2倍) |
| 💰 大入り袋 | 今すぐ軍資金 ＋80 |

## 作成の背景

`/deep-research` による市場調査(2026-06)で、**プッシュユアラック/デッキビルド/ローグライト**が `Balatro`(2024年3月に月間430万視聴時間でブレイク、ジャンル全体は四半期1170万時間ピーク)以降の最大の成長軸だと確認したのが出発点。出典: [Stream Hatchet: Indie Roguelike Deck-builders on Live Streaming](https://streamhatchet.com/blog/indie-roguelike-deck-builders-on-live-streaming/)

- 「賭け金を選ぶ→引く/止める→21への近さで配当が跳ねる」は**プッシュユアラックの最小形**。Balatro流の**指数的に上がるノルマ + ラウンド間ショップ(縁起物)**を重ねて中毒性を作った。
- 既存作には運試し系(undameshi/mezame)があるが、本作は**自分の判断(引く/止める/賭け金/縁起物の選択)が結果を左右する戦略ローグライト**で、純粋運ゲーとは別物。KAITEN(役×倍率のドラフト)とも被らない(役の概念がない押し引き)。

データは **[Deck of Cards API](https://deckofcardsapi.com/)** の公開API。6デッキをシャッフルして220枚を一括ドロー(`new/shuffle/?deck_count=6` → `draw/?count=220`)し、ローカルで配る。CORS対応・キー不要・無料。実在のトランプ画像(PNG)を使う。

> 調査メモ: Deck of Cards APIの**pile(捨て札/手札の山)管理機能は検証で確認できなかった**ため、本作は基本の `shuffle`/`draw` のみで実装している(pile非依存)。

## 技術構成

- 単一HTML(外部ライブラリなし)。手札はDOM、チップ粒子とタイマーは `requestAnimationFrame` 1本で駆動。
- デッキは起動時にAPIから6デッキ220枚を取得して使い回し、残り12枚を切ると裏で補充。**通信失敗時はローカルで6デッキ生成 + CSS描画カード**にフォールバックし、無言で壊れない。
- カード画像は `onerror` でCSS描画カード(ランク+スート)に自動フォールバック。
- エースのソフト/ハード判定、5枚チャーリー、ナチュラル(2枚21)を実装。勝敗判定はバースト・親バースト・大小・引き分けの標準ロジック。
- iOS消音スイッチ対策(無音WAV + `resume()` + `onstatechange`)。効果音はWeb Audioでリアルタイム合成。
- スマホ対策: viewport固定、セーフエリア、Pointer Events、ダブルタップズーム無効、`max-width: min(480px,64vh)` の縦カラムでPCワイド画面対応 + キーボード操作。
- OGP一式(絶対URL)。`assets/ogp.jpg` は 1200×630。

## テスト

1. `node --check` で構文チェック。
2. puppeteer-core + ローカルChrome + モバイルビューポートでの**勝敗ロジック検証E2E**: 42ハンドを自動プレイし、画面の手札合計とトースト表記を突き合わせて「勝ち/負け/引き分け」の判定が一致するか検証 → **不整合0件**。7回の場クリア(縁起物選択)も通過。
3. 全額ベットbotで**破産→ゲームオーバー画面**、**縁起物選択画面**の描画も確認。`pageerror`/console エラー0(favicon除く)。
4. ローカルHTTPサーバ(`127.0.0.1`)経由でテスト。

## 開発で得た教訓

- ブラックジャックは「ただの運ゲー」に見えるが、**配当を21への近さでスケール**させるだけで、毎手「あと1枚引くか」の押し引きが生まれ、Balatro的な貪欲のループになる。
- E2Eで**勝敗判定そのものを画面表示と突き合わせて自動検証**しておくと、配当・倍率の改修でロジックを壊しても即気づける。トランプ系は特に有効。
