# タネトリ (TANETORI)

落ちてくるマメをタネで撃ち落とすレトロ風アクション。任天堂『鳥とマメ』収録の「Pyoro 2」(GBA『メイド イン ワリオ』のミニゲーム)のゲームメカニクスを参考にしたファンメイドのオマージュ作品。グラフィック・サウンド・名称はすべてオリジナル。

- **プレイURL**: https://nrtkumi.github.io/games/tanetori/
- **対応環境**: スマホ(縦画面・タッチ)優先。PCはキーボード(←→ / A・D移動、Space / Z / ↑発射)でも遊べる

## 遊び方

- ◀▶ボタンでトリを左右に移動、「タネ発射」ボタンで向いている方向のナナメ45°上にタネを撃つ(即着弾、射線上のマメを全て同時破壊)
- マメをまとめて撃つほど高得点
- マメが床に落ちるとその列の床がこわれる。穴の上は通れない(胴体半分で停止)
- トリにマメが当たるとゲームオーバー
- スコアが上がると落下速度・同時出現数(最大16)が増え、背景が昼→夕→夜→セピア→モノクロ→花火と変化

### 得点表

| 同時撃破数 | 1個あたりの得点 |
|---|---|
| 1個 | 50 |
| 2個 | 各100 |
| 3個 | 各300 |
| 4個以上 | 各1000 |

### 特殊マメ

| 種類 | 出現 | 効果 |
|---|---|---|
| 白マメ | 約1/9の確率 | 撃つと天使が降りてきて一番近い壊れた床を1枚修復(壊れた床がなければ+200) |
| 七色マメ | 5000点で初出現、9000点以降は1000点ごと | 撃つと画面上のマメ全消し(各+50)+天使が最大10体来て床を修復 |

## 作成の背景

ユーザーリクエスト「鳥とマメ2のクローン」に基づき、原作仕様をWebで調査して再現した。主な出典:

- [Pyoro 2 - Super Mario Wiki](https://www.mariowiki.com/Pyoro_2) — 種は即着弾で射線上を全破壊、コンボ得点(50/100/300/1000)、点滅マメの出現タイミング(5000点、9000点以降は1000点ごと)、葉っぱ演出
- [Pyoro (minigame) - Super Mario Wiki](https://www.mariowiki.com/Pyoro_(minigame)) — 床20列、白マメ(1/9)で天使が床修復、点滅マメで全消し+天使最大10体、マメ落下で床破壊、トリ被弾でゲームオーバー、同時出現最大16、スコアによる背景変化(夕→夜→セピア→モノクロ→花火)
- [鳥とマメ - 任天堂](https://www.nintendo.co.jp/ds/dsiware/kp6j/index.html) — 原作の基本ルール

## 技術構成

- 単一HTML(canvas 2D + Web Audio リアルタイム合成)+ 生成画像アセット5点
- **画像アセット**(Codex CLI 経由の gpt-image-2 で生成):
  - `assets/bg.jpg` — 背景の風景イラスト(昼の空)。スコアフェーズごとの色調変化は、この1枚に乗算合成(`multiply`)でティントを掛けて表現(夕=オレンジ、夜=青、セピア、モノクロは`saturation`合成で脱色、花火=紫)。読み込み失敗時はグラデーション描画にフォールバック
  - `assets/logo.png` — タイトルロゴ。gpt-image-2は透過背景非対応のため、白背景で生成→縁からのflood fill+1pxフェザーで透過化(文字内の白ハイライトは保持)
  - `assets/ogp.jpg` — OGP/シェア画像(og:image / twitter:card メタタグ設定済み)
  - `assets/bird.png` / `assets/bird_shoot.png` — 操作キャラのスプライト(待機/発射の2ポーズ)。1枚の横長画像に2ポーズ並べて生成→左右分割→flood fill透過化→不透明部分のbboxで自動クロップ。左向きは `ctx.scale(-1,1)` で反転、移動時のホップはY座標で表現。画像未ロード時は従来のcanvas描画にフォールバック
  - いずれも sips でリサイズし計約700KBに最適化
- マメ・天使・葉っぱ・パーティクルはcanvas描画のまま(色変化・発光・回転が自由なため)
- スマホ対応: viewport-fit=cover + safe-area-inset、touch-action: none、Pointer Events統一、マルチタッチ(移動しながら発射)対応
- ダブルタップズーム無効化: iOS Safariは`user-scalable=no`を無視するため、全要素`touch-action: manipulation` + `dblclick`/`gesturestart`/連続`touchend`(350ms以内)のpreventDefaultで抑止。touchend抑止で合成clickが消えるためボタン類はPointer Eventsで処理
- iOS消音スイッチ対策: 初回タップで無音WAVを `<audio>` ループ再生(オーディオセッション切替)+ `AudioContext.resume()` のPromise完了待ち + `onstatechange` での自動復帰(games/hakai と同方式)
- メインループ(rAF)はタイトル画面から常時稼働、ループが触る状態は全て宣言時に初期化
- ベストスコアは localStorage(try/catchフォールバック付き)
- フォント: Bungee + Zen Kaku Gothic New + Space Mono(サイト共通)

## 実施したテスト

1. **構文チェック**: インラインscriptを抽出して `node --check` — 通過
2. **ヘッドレスロジックテスト**(DOMスタブ+Node): タイトル時点のループ稼働 / 開始 / 2個コンボ=+200 / 白マメ→天使が床修復 / 七色マメ→全消し得点 / マメ落下→床破壊 / トリ被弾→ゲームオーバー / ベスト保存 / リトライ — 全9項目通過
3. **実ブラウザE2E**(puppeteer-core + ローカルChrome、390×844・hasTouch): ロード時エラーゼロ / canvas描画ピクセル確認 / 背景画像ロード確認 / タップ開始 / ▶長押しで移動 / 発射でスコア加算 / 5秒以上の自動プレイでconsoleエラーゼロ / ゲームオーバー→リトライ / localStorage永続化 / スクリーンショット目視確認 — 全通過
4. **フェーズ別ビジュアル確認**: デバッグフックでスコアを5000/10000/30000/42000に設定し、夕方・夜・モノクロ・花火の各ティントをスクリーンショットで目視確認
5. **スプライト切り替え確認**: puppeteerのclip付きスクリーンショットでトリ周辺をクローズアップし、待機/発射(くちばし開き)/左向き反転の3状態を目視確認

### 開発で得た教訓

- ヘッドレステストでゲーム内オブジェクトを検証するときは、**自然スポーンする敵(マメ)を毎フレーム除去**しないとテストが不安定になる(天使の修復先が意図しない列にズレる等)
- 発射クールダウンのような時間依存処理は、テスト側で「大きな1フレーム」を送ってもdtクランプ(0.05s)で消化されない。複数フレームに分けて進める
