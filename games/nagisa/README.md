# ナギサパーク — A Day at Nagisa

海辺のテーマパークの、忘れられない一日。**閉園までの限られた時間**で、刻々と変わる**待ち時間**を読みながらテーマポートを巡り、アトラクションを乗り継いで連鎖倍率「**ノリ**」を高め、疲れて**バスト**する前に休憩で確定させ、夜の**ハーバーショー**で一気に放出して最高の思い出を作る——**ルート最適化 × push-your-luck ローグライト**。

- **プレイURL**: https://nrtkumi.github.io/games/nagisa/ （※下記「公開について」を参照）
- **対応環境**: スマホ（縦画面・タッチ操作）推奨。PC（マウス/キーボード不要、タップ相当のクリック）でも動作
- **セッション時間**: 1プレイ 約3〜6分（パークの「一日」= 9:00〜21:00）

---

## 遊び方

1. **相棒をえらぶ**：パッシブ効果を持つ相棒を1匹選んで入園。
2. **ポートをタップ**して移動 → そのポートのアトラクションパネルが開く。
3. **アトラクションに乗る**と「未確定の思い出」がたまり、乗るたびに**ノリ（連鎖倍率）が上昇**。同じテーマを連続で乗ると伸びが大きい（🔥テーマ連鎖）。
4. ただし**元気が45%を切ると、乗るたびにバストの危険**。バストすると「未確定の思い出」を**すべて失い、ノリも×1.0にリセット**。
5. **グルメ／休憩**で「未確定」を**確定（スコア化）**し、元気を回復。ただしノリは×1.0に戻る。
   - → 「安全に確定させる」か「ノリを伸ばして温存し、もっと大きく狙う」かの駆け引きが核。
6. 夕方以降、中央の**霧の火山島**で**ハーバーショー**が解禁。観ると**未確定の思い出を大きな倍率で放出**してフィナーレ（最高ノリが高いほど倍率UP）。観ると一日が終わる。
7. 閉園（21:00）までに、どれだけ大きな「思い出」を作れるか。

## 作成の背景

「テーマパークの一日」を題材に、何を“面白さの核”に据えるかを `deep-research`（複数ソースのファンアウト検索＋出典付き検証）で調査した。検証済みの要点：

- **1日のルート／待ち時間の最適化**は実在の有望フック。乗る順番で待ち時間を最小化する park-day 計画アプリ（RideReady）は「park dayを丸ごと計画し、30分ごとに再計画、待ち時間が60秒ごとに更新される」ことを売りにしている。→ **混雑で揺れる待ち時間に合わせて巡る順を組み替え続ける**のがエンゲージの核（出典: https://rideready.app/ ）。
- **ゲストの気分（満足度）がセッション長に直結**：気分が良いほど長く滞在し、悪いと帰ってしまう（出典: https://www.planetcoaster.com/news/2024-09-25/deep-dive-mastering-management ）。→ 本作の「元気→楽しめる量」「疲れでバスト」に反映。
- 短セッション・中毒性のローグライト要素として、**段階的に上がるターゲット**と**push-your-luck / シナジー構築**（Balatro、Luck Be a Landlord、Dogpile などのデッキ/シンボル系で繰り返し言及）。→ 本作の**ノリ連鎖**と**バスト**、**ショーでの倍率放出**に反映。

これらを、東京ディズニーシーの体験構造である「**複数のテーマポート＋中央のハーバーと火山＋夜のハーバーショー**」に重ねてオリジナル世界観で再構成した（地中海風の港町／アラビアン風の市場／レトロフューチャーの波止場／密林の遺跡／海底ラグーン／ノスタルジックな灯台波止場＋中央の火山島）。

### デザインコンセプト

**「1930年代のトラベルポスター × 古い海図（nautical chart）」**。羊皮紙のような島、ディープティールの海に彫版風の波線ハッチング、真鍮ゴールドと珊瑚レッドの差し色、点線の航路、Cinzel（彫り文字風）＋ Shippori Mincho B1（レトロ明朝）＋ Zen Kaku Gothic New。生成AIが作りがちな紫グラデ／グラスモーフィズム／ネオングローは避け、「世界一周クルーズ」のロマンを当世風に。

## 技術構成

- **単一HTMLファイル**（外部依存はGoogle Fontsのみ）。ゲームロジック・描画・音は全てインライン。
- **描画**: Canvas 2D。ハーバーマップ（六角形に並ぶ6ポート＋中央の火山島、点線航路、波線ハッチング、待ち時間バッジ、火山のショー演出）を `requestAnimationFrame` で常時描画。**ループはタイトル時点から回り、状態は宣言時に全初期化**（初回フレーム例外でループが死ぬ事故を防止）。DPR変換は**毎フレーム `ctx.setTransform`** で再設定（iOSのコンテキスト喪失対策）。
- **ゲーム進行**: ターン制（イベント駆動）。タップ→パネル→行動で時間が進み、待ち時間が再抽選される。
- **音**: Web Audio のリアルタイム合成（外部音源なし）。**iOS消音スイッチ対策**として、初回タップで無音WAVを `<audio loop>` 再生＋ `AudioContext.resume()` の Promise 完了待ち＋ `onstatechange` の `interrupted` 復帰を実装。
- **入力**: Pointer Events に統一（`click` 非依存）。ダブルタップズーム無効化（`touch-action`、`dblclick`/`gesturestart` preventDefault、350ms以内の連続 `touchend` 抑止）。`viewport-fit=cover` ＋ `env(safe-area-inset-*)`、PCワイド対策に `max-width` で縦長カラム中央寄せ。
- **永続化**: localStorage（try/catch フォールバック）。ベストスコア・収集フレーバー・解放済み相棒。
- **OGP画像**: 縦のプレイ画面を puppeteer でスマホ枠カード（1200×630）に合成して生成（`assets/ogp.jpg`）。

## 実施したテストと教訓

- **構文チェック**: インラインスクリプトを抽出して `node --check`。
- **ヘッドレスロジック / 実ブラウザE2E**: puppeteer-core + ローカルChrome、モバイルビューポート（`hasTouch`）＋実タッチで、タイトル→入園→ポートタップ移動→乗車→ショー→閉園→結果→図鑑まで通し。**`pageerror`/console エラーがゼロ**、canvas に描画ピクセルがあること、状態が進むことを確認。`file://` ではなく `python3 -m http.server` 経由で実行（同梱画像によるcanvas汚染で `getImageData` が SecurityError になるのを回避）。
- **バランス調整（ボット約3000ラン×複数方策）**: 完全情報の貪欲ボットでスコア分布を測定して調整。
  - 当初は**中スタミナ帯のバスト率が高すぎ**（〜40%が運任せで理不尽）→ **元気45%以上はバスト0%、下回ると押し込むほど危険**に変更。「疲れてきたが、もう一本いくか?」という**自分で選ぶ賭け**に。ボタンに⚠バスト率を表示して情報を開示。
  - ショーの**固定ボーナスがスコアを平坦化**していた→ **倍率主体（最高ノリに比例）**に寄せ、固定値を圧縮。これにより「安全に確定」と「温存してショーで放出」の駆け引きが実際に拮抗（show-saver方策がgreedy方策を必ずしも上回らない=健全な賭け）ことを確認。
  - **ランクと相棒の解放条件を実分布に合わせて再設定**（当初は最大スコアの10倍近い閾値で“死にコンテンツ”だった）。教訓どおり、完全情報ボットは実プレイより1〜2割甘いので、上位ランクはボットを上回る熟練で届く位置に置いた。
