# ドアップめし (DOAPPU MESHI)

超ドアップにズームされた**実在の料理写真**が約8秒かけてゆっくりズームアウト。
何の料理かわかった瞬間に4択から名前をタップする早押しクイズ。早く当てるほど高得点(100〜1000pt)。
出題はウィキペディアの料理記事のリード画像から毎回ランダムに変わる。[ドアップ!](../doappu/)(生き物版)の料理版。

- **プレイURL**: https://nrtkumi.github.io/games/doappu-meshi/
- **対応環境**: スマホ(縦画面・タッチ)/ PC(1〜4キー・マウス)
- **通信**: 必要(ウィキペディアの公開APIをブラウザから直接叩く)
- **デザインコンセプト**: 昭和の食堂・食品サンプルのショーケース(クリーム地+トマトレッド/マスタード、ドット文字の電光カウンタ、ハードシャドウのシール質感)

## 遊び方

1. 品目の開始時に「つぎは…🍜 麺類!」のようにジャンルが予告される
2. 超ドアップ(8倍)の料理写真が約8秒かけてゆっくりズームアウトしていく
3. 何の料理かわかったら下の4択から名前をタップ。**残りズーム率が大きいほど高得点**
4. 答え合わせで写真全体+料理名+ジャンル+出典(ウィキペディア記事へのリンク)を表示。全10品・ライフ3

| 要素 | 内容 |
|---|---|
| スコア | 回答時点の残りズーム率に応じて 100〜1000pt(ズームアウト完了後でも回答可・最低100pt) |
| コンボ | 連続正解で +50pt × 連続数 のボーナス |
| ライフ | 不正解で ❤️ -1(正解は表示される)。ライフ0で終了 |
| ジャンル | 和食・ご飯/丼・麺類・中華・洋食/西洋・アジア/エスニック・スイーツを品目ごとに切り替え |
| ダミー選択肢 | 同じジャンルの別の料理名(似たもの同士で迷わせる) |
| リザルト | スコア・正解数・評価コメント・ベスト保存(localStorage)・X シェア |

## 作成の背景

[ドアップ!](../doappu/)(生き物のズームアウト早押しクイズ)の姉妹作。コア体験はそのままに、被写体を「実在の料理写真」に置き換えた料理版。

市場調査の要点(姉妹作と共通):

- 2026年のブラウザゲームはハイパーカジュアルがプレイ時間シェア78.8%を占め、「最初の10秒でルールが分かる」ことがヒットの条件([primepulselogic](https://primepulselogic.com/browser-games-2026-viral-trend/))
- ブラウザゲームシーンはNYT Games系のデイリー/クイズ系とindie HTML5が主流([Dinogame GG](https://dinogame.gg/blog/best-new-browser-games-2026/))

「画像の段階的リビール」はテレビのズームクイズ番組や Framed / Pokedoku 的なデイリーパズルの系譜で実績のあるフォーマット。そこに「いつ答えるか=スコアを賭けたチキンレース」という早押しのリスクリターンを足したのがコア。料理は「茶色いソースの寄り(カレー/麻婆/ハヤシ?)」のように寄ると正体が曖昧になり、同ジャンルのダミーで迷わせられるので、見慣れた料理でもズームクイズが成立する。実在写真を使うため出題が毎回変わってコンテンツが古びない。

## 技術構成

- 依存ライブラリなしの単一HTML(DOMベース、写真のズームはCSS transform + rAF駆動)
- **使用API(キー不要・無料・CORS対応)**: ウィキペディア(日本語版)`GET /w/api.php?action=query&prop=pageimages&piprop=thumbnail&pithumbsize=1000&pilimit=50&titles={最大50件を|区切り}&redirects=1&formatversion=2&origin=*`
  - 料理マスタは**86品をキュレーション**(7ジャンル)。各品の「記事タイトル(画像が確実に取れる解決済みタイトル)」と「表示名」を分けて保持し、`豚カツ→とんかつ`・`カスタードプディング→プリン`・`パットガパオ→ガパオライス` 等を表示名で見せる
  - 記事のリード画像サムネ(`thumbnail.source`)を `upload.wikimedia.org` から直接取得。`normalized`/`redirects` を解決してリクエストタイトル→最終タイトル→サムネをマッピング
  - 全86件を**2リクエスト(50件+36件)でまとめて取得**してジャンル別プールを構築。サムネが無い記事は出題から除外し、4択が組めないジャンルも除外
  - 出題写真は `Image()` でプリロードし、ロード失敗した料理はスキップして次候補へ。ダミーは同ジャンルの別料理名から3つ
- ズーム演出: `scale(8)`→`scale(1)` を8秒、`Math.pow(8, 1-p)` で知覚的に等速なズームアウト。`transform-origin` は写真ごとにランダム(12〜88%)で毎回違う部位から始まる
- 次の品はリビール中に裏でプリフェッチ(写真プリロード)して待ち時間ゼロに
- 効果音はWeb Audioでリアルタイム合成(ズーム中の加速するチック音、正解のアルペジオ、不正解ブザー)。iOS消音スイッチ対策(無音WAVを`<audio>`でループ再生、`AudioContext.resume()`のPromise完了待ち、`interrupted`からの自動復帰)
- 入力はPointer Eventsで統一(`click`不使用)、ダブルタップズーム無効化3点セット、PCワイド画面は縦長カラム(`min(100vw, 520px, 64vh)`)に制限
- ベストスコア・ミュート設定はlocalStorage(try/catchでフォールバック)
- フォント: タイトル RocknRoll One / 本文 Zen Maru Gothic / 数値 DotGothic16(電光カウンタ風)
- OGP画像はpuppeteerでミッドズームの実プレイ画面を撮り、1200x630のスマホ枠カードにブラウザ内合成(外部API課金なし)
- クレジット表記: タイトル画面に「写真・解説: ウィキペディア / ウィキメディア・コモンズ(各種CCライセンス)」、各答え合わせに出典記事へのリンクを表示

## 実施したテスト

1. **構文チェック**: インラインscript抽出 → `node --check` パス
2. **ヘッドレスロジックテスト(Node、DOM/fetch/Image/rAFスタブ+合成サムネ)**: 全20チェックパス
   - 開始→ズーム進行→早押しスコア計算(2秒時点で中得点)→コンボ加算→10品完走→ベスト保存→localStorage永続化
   - 全問不正解→3ミスでGAME OVER・ライフ0・正解数0、低スコア時はベスト不変、タイトル復帰
3. **実ブラウザE2E(puppeteer-core + ローカルChrome、390x844・hasTouch、実API、ローカルHTTPサーバ:8779経由)**: 全11チェックパス
   - 実APIから取得した `upload.wikimedia.org` の写真(naturalWidth=1280)で10品通しプレイ(タッチ操作のみ、完全情報で10/10・10715pt)
   - ズームtransform適用・早押しスコア・リザルト・ベスト保存を確認
   - `pageerror`・consoleエラー 0件、スクリーンショットで見た目を目視確認

## 開発で得た教訓

- ウィキペディアの `prop=pageimages` は **`pilimit=50` を付けないと複数タイトル指定でも先頭1件しかサムネが返らない**ことがある。`formatversion=2` だと `query.pages` が配列で扱いやすい
- 料理名は記事タイトルと表示名がズレる(`豚カツ`/`カスタードプディング`/`パットガパオ` 等)。**クエリ用タイトルと表示名を分離**し、redirects を解決して結びつけると、見せ方は自然なまま実画像を確実に取れる
- リード画像は基本どの料理記事でも「料理そのものの写真」が来るので、生き物版(iNaturalist)の和名欠損のようなヒット率問題がなく、キュレーションした86品はほぼ全数が有効だった
- 料理は寄ると正体が曖昧になりやすく(エビチリ/麻婆豆腐の赤い寄り等)、同ジャンルのダミーと相まって見慣れた題材でもズームクイズとして成立する
