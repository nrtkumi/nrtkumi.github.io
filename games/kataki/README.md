# KATAKI — 蚊叩き

カメラの前で手を振って、画面を飛び回る蚊を叩き潰すモーションアクション+ウェーブ制ローグライト。

- **プレイURL**: https://nrtkumi.github.io/games/kataki/
- **対応環境**: スマホ(縦画面)・PCブラウザ。カメラ(フロント)推奨だが、**カメラなしでもタッチ/クリックでプレイ可**。依存ライブラリなしの単一HTML
- **プライバシー**: カメラ映像は端末内のフレーム差分解析にのみ使用。描画と解析はすべてローカルで完結し、**保存・送信は一切しない**

## 遊び方

1. 「📷 カメラで叩く」をタップしてカメラを許可(嫌なら「カメラなし」でタッチモード)
2. 自分が映った画面の上を蚊が飛び回るので、**蚊が見える場所で実際に手を振る**と叩き潰せる(タッチ/クリックでも叩ける)
3. 蚊は放置すると着地して**吸血開始**(リングタイマー)。吸われきるとHP−1、HP0でゲームオーバー
4. 🐝 **蜂を叩くとシビれて**1.3秒間何もできず、コンボも切れる。蜂はスルーが正解
5. 🌀 **蚊取り線香**が出たら叩くと数秒間スローモーション
6. ウェーブの規定数を叩くとクリア。**3択から強化を1つ選んで**次のウェーブへ
7. 連続キルでコンボ倍率アップ。ベストスコアは端末に保存

## 作成の背景(市場調査の要点)

2026年6月時点の調査をもとに「カメラ×体感」を採用した。

- **カメラモーション操作がメインストリーム化**: 『ジャストダンス2026エディション』がスマホカメラで全身の動きを検出するカメラコントローラー機能を正式リリース([Ubisoft](https://www.ubisoft.com/ja-jp/game/just-dance/2026/news-updates/2UBGhF4xWMLAVVFMmTCWwn))。ポーズ推定ベースのWebカメラフィットネスゲームも台頭([67 Speed Games](https://67speedgames.com/blog/webcam-fitness-games-future/))
- **「手を振って画面に触れる」はEyeToy以来の定番で5秒で理解できる**: WebCam Mania等のWebカメラゲームはフレーム差分のモーション検出だけで成立しており、ML不要・軽量でブラウザ向き([WebCam Mania](https://webcam.sulat.net/)、[itch.io webcamタグ](https://itch.io/games/tag-webcam))
- **ハイブリッドカジュアルが2026年の主流**: 10秒で分かるコアメカニクスに進行・強化のメタレイヤーを重ねる構成([Game Growth Advisor](https://gamegrowthadvisor.com/blog/2026-04-16-hybrid-casual-game-design-strategy-2026/)、[Udonis](https://www.blog.udonis.co/mobile-marketing/mobile-games/casual-games))→ ウェーブクリアごとの3択強化を採用
- サイト内の既存13ゲーム(タイミングタップ、ブロック崩し、サバイバー、マージ等)にカメラ・体感系はなく、ジャンルの空きを埋める

「蚊を叩く」は誰でも体が覚えている動作で、チュートリアル不要。6月(蚊の季節)のリリースに合わせた季節ネタでもある。

## 技術構成

- 依存ライブラリなしの単一HTML(canvas + Pointer Events + getUserMedia)
- **モーション検出はフレーム差分**: フロントカメラ映像を表示と同じミラー&cover-fitで低解像度キャンバス(幅90px・高さは画面比に追従)に毎フレーム描き、輝度差分を9×12グリッドに集計。蚊の周囲セルのモーション強度がしきい値を超えたら「叩いた」判定。MLモデル・外部ライブラリ不使用
  - **全体モーションガード**: 画面全体が動いた場合(カメラを持って歩いた等)は誤判定としてヒット無効。局所モーションが全体平均を十分上回ることも要求
  - 解析キャンバスを表示と同じ座標系にしておくことで、画面座標→グリッドの対応が単純な比例計算で済む
- **カメラ不許可・非対応時はタッチモードへ自動フォールバック**(部屋のイラスト背景に切り替え)。カメラモード中もタッチ/クリックで叩ける
- 効果音はWeb Audioでリアルタイム合成。蚊の羽音は残存数に応じた持続ノコギリ波+ビブラートLFO
- **iOS消音スイッチ対策**: 初回タップ時に無音WAV(JSでArrayBuffer生成→Blob URL)を`<audio>`でループ再生、`AudioContext.resume()`のPromise完了待ち、`onstatechange`で`interrupted`から自動復帰(games/hakai/と同方式)
- メインループ(rAF)はスタート画面から回り、ループが触る状態は全て宣言時に初期化。dpr変換は毎フレーム`ctx.setTransform`で再設定(コンテキスト喪失対策)
- ベストスコアはlocalStorage(try/catchでフォールバック)
- OGP画像はCodex CLI(gpt-image-2)で生成したキービジュアル(夜の部屋+漫画風の蚊+迫る手のひら+タイトルロゴ、1536×1024で生成→1200×630にクロップ)
- テスト用に読み取り+注入フック`window.__KATAKI__`を公開(phase/score/bugs/モーション注入など)

## 実施したテスト

1. **構文チェック**: インラインscriptを抽出して `node --check` → OK
2. **ヘッドレスロジックテスト**: DOM/canvas/localStorageをスタブしてNode(vm)でrAFを手動駆動。スタート画面でのループ生存→タッチモード開始→スポーン→叩いてスコア/キル加算→ウェーブクリア→強化3択→選択→蜂はキル数に入らない(シビれ)→吸血でHP減→ゲームオーバー→ベスト保存→リトライ、全18項目PASS
3. **実ブラウザE2E**: puppeteer-core + ローカルChrome(`--use-fake-device-for-media-stream`でフェイクカメラ)、モバイルビューポート(390×844、`hasTouch: true`)+タッチ操作、ローカルHTTPサーバ経由。pageerror/consoleエラーゼロ、canvas描画ピクセルあり、**カメラ起動と映像再生**、**フレーム差分のモーション解析が実映像で稼働**(フェイクカメラの動くパターンをglobalMotionとして検出)、タッチで複数キル、モーション注入経由のキル、ウェーブクリア→強化選択、ゲームオーバー→localStorage保存→リトライ、全17項目PASS。スクリーンショットで見た目も目視確認

### 開発で得た教訓

- **モーション解析キャンバスは表示キャンバスと同じアスペクト比・同じミラー&cover-fitで描く**と、画面座標→解析グリッドの変換が単純比例で済み、ズレない。解析解像度は幅90px固定で高さを画面比に追従させた(リサイズ時は前フレームバッファを破棄)
- ChromeのフェイクカメラはE2Eに有用: `--use-fake-ui-for-media-stream`(許可ダイアログ自動承認)+`--use-fake-device-for-media-stream`(動くテストパターン)で、getUserMedia〜フレーム差分検出までを自動テストできる
- E2Eで「一定間隔でテスト用の状態リセットフック(forceBitePhase等)を呼び続ける」と、進行タイマーが毎回リセットされて永遠に完了しない。**フックは1回呼んで完了時間ぶん待つ**形にする
- 実写映像の上に描くゲームオブジェクトは背景との明度差が保証されないので、淡い光彩(背後に半透明の円)を敷いて視認性を確保する
