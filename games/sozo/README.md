# SOZO 8192 — 宇宙創造パズル

塵を重ね、星を紡ぎ、宇宙を創れ。あなたは虚無に座す見習いの創造神。
6x6の盤面で同じ天体をマージし、塵(2)から宇宙(8192)の創造を目指すスライドパズル。

- **プレイURL**: https://nrtkumi.github.io/games/sozo/
- **対応環境**: スマホ(タッチ・縦画面)/ PC(キーボード ←↑↓→ / WASD)

## 遊び方

- スワイプ(または矢印キー)で盤面の天体全体をスライド
- 同じ天体がぶつかると合体して1段階進化し、スコアが入る
- 移動のたびに新しい天体(塵か礫)が**2つ**出現する
- 動かせなくなったらゲームオーバー。**宇宙(8192)** を創れば勝利(その先の多元宇宙へ続行も可能)

### 天体の進化(13段階)

| 数値 | 天体 | 数値 | 天体 |
|---|---|---|---|
| 2 | 塵 | 256 | 恒星 |
| 4 | 礫 | 512 | 超新星 |
| 8 | 岩塊 | 1024 | 中性子星 |
| 16 | 彗星 | 2048 | ブラックホール |
| 32 | 月 | 4096 | 銀河 |
| 64 | 惑星 | 8192 | **宇宙(勝利)** |
| 128 | 巨星 | 16384 | 多元宇宙(続行) |

### 脳汁システム

- **連結(COMBO)**: 1手で複数マージすると COMBO 表示+音程上昇
- **連鎖(チェーン)**: マージを含む移動を続けるとゲージが貯まりやすくなる
- **創造の刻(フィーバー)**: ゲージ満タンで12秒間スコア2倍。星空がワープし盤面が金色に発光
- **天体誕生バナー**: その回で初めての天体が生まれると名前入りバナーで祝福
- 段位に応じてパーティクル・衝撃波・画面シェイクが激化。512以上はヒットストップ+全画面フラッシュ、2048はブラックホールの吸い込み、8192はビッグバン演出

## 作成の背景(市場調査)

- 8192は2048の上位バリアント。**6x6グリッド・毎ターン2タイル出現・8192タイルで勝利**という仕様が定番
  ([play-2048.com/8192](https://www.play-2048.com/8192/), [poweroftwo](https://poweroftwo.nemoidstudio.com/8192))
- マージ系はカジュアルパズルの第2勢力に成長しており、グリッドベースの簡素なマージが広い層に刺さる
  ([Gamigion: Casual Puzzles in 2026](https://www.gamigion.com/casual-puzzles-in-2026-the-past-the-present-and-the-future/))
- 「ゲームジュース」(即時フィードバック・誇張エフェクト)がプレイヤー満足度の核という設計論を全面採用
  ([The Design Lab: Making Gameplay Irresistibly Satisfying Using Game Juice](https://thedesignlab.blog/2025/01/06/making-gameplay-irresistibly-satisfying-using-game-juice/))
- サイト内の既存ゲーム(破壊系ブロック崩し、種取り、おみくじ等)とジャンル不被り。hakai(破壊)と対になる「創造」をテーマに採用

## 技術構成

- 依存ライブラリなしの単一HTML。盤面・演出・UIすべてcanvas描画
- 天体13種は外部画像なしのcanvasプロシージャル描画(カード/天体/グローの3層スプライトを事前レンダリング、決定的乱数で再現性確保)
- タイトルのキービジュアルは Codex CLI 経由の gpt-image-2 で生成(「創造神の手が銀河を包む」構図、800px JPEG化)。読み込み失敗時はcanvas描画の銀河にフォールバック
- OGP(1200x630)はキービジュアル背景+実プレイ盤面ショットをcanvasで合成
- 入力はPointer Eventsに統一(clickイベント不使用)。スワイプはpointermove中の閾値超えで即発火
- iOS消音スイッチ対策(無音WAVループ+AudioContext.resume待ち+interrupted復帰)、ダブルタップズーム3点対策
- 効果音はWeb Audioでリアルタイム合成(マージ音は段位と連結数で音程が上がる)
- dpr変換は毎フレーム `ctx.setTransform` で再設定(コンテキスト喪失対策)
- PCワイド画面はルートを `min(100vw, 520px, 64vh)` の縦長カラムに制限
- ベストスコア・最高天体・ミュート設定は localStorage(try/catchフォールバック)

## 実施したテスト

1. **構文チェック**: インラインscript抽出 → `node --check`
2. **ヘッドレスロジックテスト(27項目)**: DOMスタブでマージ規則(1手1マージ、4連→2ペア、再マージ禁止)、移動可否判定、詰み判定、スコア/連鎖/ゲージ、600手自動プレイ、永続化を検証
3. **実ブラウザE2E(15項目)**: puppeteer-core + ローカルChrome、モバイルビューポート(hasTouch)でタッチスワイプ操作。タイトル→プレイ→512生成→フィーバー突入→8192ビッグバン→勝利→続行→詰みゲームオーバー→リスタートまで通し、コンソールエラーゼロとcanvas描画ピクセルを確認
4. **デスクトップE2E**: 1440x900でEnter開始+矢印/WASD操作を確認

### 開発で得た教訓

- `file://` でテストすると同梱画像でもcanvasが汚染され `getImageData` が失敗する。実ブラウザE2Eは最初からローカルHTTPサーバ経由で行う
- 同じ固定座標に出すフロートテキスト(COMBO表示)は連発時に重なって文字化けのように見える。出現位置に揺らぎを入れる
