# ペディアバトル (PEDIA BATTLE)

2つのWikipedia記事のうち、**先月多く読まれたのはどっち?** を当てる連勝クイズ(Higher/Lower系)。
Wikimedia Pageviews API の実データを使い、出題は日本語版Wikipediaの先月の閲覧数トップ1000記事から毎月自動更新される。

- **プレイURL**: https://nrtkumi.github.io/games/pedia-battle/
- **対応環境**: スマホ(縦画面・タッチ)/ PC(↑↓キー・マウス)
- **通信**: 必要(Wikipedia/Wikimediaの公開APIをブラウザから直接叩く)

## 遊び方

1. 上のカード(王者)には記事と**先月の閲覧数**が表示される
2. 下のカード(挑戦者)の閲覧数が上より「**▲ 多い**」か「**▼ 少ない**」かを選ぶ
3. 正解すると挑戦者が新たな王者になり、次の挑戦者が現れる。連勝を伸ばせ
4. 不正解で即ゲームオーバー(ベスト連勝は端末に保存)

| 要素 | 内容 |
|---|---|
| しおり 🔖 | 5連勝ごとに1個獲得(最大2)。ミスを1回だけ帳消しにして続行できる |
| 難易度カーブ | 連勝が伸びるほど閲覧数の差が小さい「接戦カード」が出る(序盤は2.5倍以上の差、20連勝以降は差なし制限) |
| 答え合わせ | リザルトに両記事の閲覧数と記事リンクを表示 |
| シェア | リザルトから X(Twitter)に連勝数をシェア |

## 作成の背景

「無料で使えるかつブラウザから叩ける公開APIを利用したゲーム」というお題から、キー不要・CORS対応(`Access-Control-Allow-Origin: *`)で叩ける Wikimedia 系APIを選定。市場調査の要点:

- 2026年のブラウザゲームはハイパーカジュアルがプレイ時間シェア78.8%を占め、「最初の10秒でルールが分かる」ことがヒットの条件([primepulselogic](https://primepulselogic.com/browser-games-2026-viral-trend/))
- 検索トレンドでも Block Blast・spacebar clicker など「説明不要の即遊べる系」が急伸([Rising Trends](https://www.risingtrends.co/trends/gaming-trends-2026))
- ブラウザゲームシーンはNYT Games系のデイリー/クイズ系とindie HTML5が主流([Dinogame GG](https://dinogame.gg/blog/best-new-browser-games-2026/))

「The Higher Lower Game」(Google検索数の多寡を当てる)で確立された比較クイズの型を、日本語版Wikipediaの閲覧数に置き換えた。実データならではの「えっ、この記事そんなに読まれてるの!?」という驚きがコア体験。出題元データが毎月更新されるため、コンテンツが古びない。サイト内の既存ゲーム(アクション、マージパズル、音楽クイズ、おみくじ、サバイバー等)とジャンルが被らない比較クイズ枠。

## 技術構成

- 依存ライブラリなしの単一HTML(DOMベース、canvas不使用)
- **使用API(いずれもキー不要・無料・CORS対応)**
  - 閲覧数トップ1000: `https://wikimedia.org/api/rest_v1/metrics/pageviews/top/ja.wikipedia/all-access/{年}/{月}/all-days` — 出題リストと答え(月間閲覧数)がこの1リクエストで揃う
  - 記事の説明文・サムネイル画像: `https://ja.wikipedia.org/api/rest_v1/page/summary/{記事名}` — カード表示用。次の挑戦者はリビール演出中に裏でプリフェッチ
- 先月分が未集計の場合は前々月までフォールバック。トップリストはlocalStorageに月単位でキャッシュし、オフライン時は古いキャッシュでも遊べる
- 出題フィルタ: メインページ・特別ページ・各種名前空間(Template:、Category: 等)を正規表現で除外、露骨な性的話題の記事はNGワードリストで除外
- サムネイルは330px→640pxへの高解像度差し替えを試行(失敗時はそのまま)。画像なし記事はタイトル文字からHSLを決めたグラデーションでフォールバック
- 効果音はWeb Audioでリアルタイム合成。iOS消音スイッチ対策(無音WAVを`<audio>`でループ再生してオーディオセッションを切り替え、`AudioContext.resume()`のPromise完了待ち、`interrupted`からの自動復帰)
- 入力はPointer Eventsで統一(`click`不使用)、ダブルタップズーム無効化3点セット、PCワイド画面は縦長カラム(`min(100vw, 520px, 64vh)`)に制限
- ベスト連勝・ミュート設定はlocalStorage(try/catchでフォールバック)
- OGP画像はcodex CLI経由のgpt-image-2で生成(ダークネイビー背景+記事カード対決+タイトルロゴの構図を日本語で指示、1536x1024→1200x800 JPEGに変換)
- 出典表記: タイトル画面とREADMEに Wikipedia (CC BY-SA 4.0) / Wikimedia Pageviews API を明記。リザルトの答え合わせから各記事へリンク

## 実施したテスト

1. **構文チェック**: インラインscript抽出 → `node --check` パス
2. **ヘッドレスロジックテスト(Node、DOM/fetchスタブ)**: 全18チェックパス
   - 名前空間・NGワードのフィルタリング
   - 開始→7連勝(しおり獲得)→ミスでしおり消費(連勝維持)→再ミスでゲームオーバー→ベスト保存→リトライ
3. **実ブラウザE2E(puppeteer-core + ローカルChrome、390x844・hasTouch、実API、ローカルHTTPサーバ経由)**: 全項目パス
   - 実APIから2026年5月のトップ1000を取得して対戦成立
   - タッチ操作のみで5連勝→しおり消費→ゲームオーバー→リトライの一連を完走
   - `pageerror`・consoleエラー 0件、スクリーンショットで見た目を目視確認

## 開発で得た教訓

- Pageviews APIの「月間トップ1000」エンドポイントは、**出題リスト・答え(閲覧数)・ランクが1リクエストで全部揃う**のでクイズの土台として非常に優秀。記事ごとの追加リクエストは表示用のsummaryだけで済む
- トップリストには「メインページ」「特別:検索」などの非記事ページとアダルト記事が必ず混ざるので、名前空間正規表現+NGワードのフィルタは必須
- カード差し替え時に高解像度画像の遅延ロードが前のカードに当たらないよう、`onload`時に「カードのタイトルがまだ同じ記事か」を確認してから差し替える
