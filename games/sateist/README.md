# サテイスト（SATEIST）

英国の **“本物の成約物件”** を、地図と物件情報から「いくらで売れた？」と査定するハイスコア・ゲーム。物件データは英政府（HM Land Registry）の**実際の成約価格記録**、位置は**実在のポストコード**、地図は**本物のOpenStreetMap**。生成データではなく、すべて実在の家です。

- **プレイURL**: https://nrtkumi.github.io/games/sateist/
- **対応環境**: スマホ（縦持ち・タッチ）優先。PCはマウス操作可。縦画面・1人用・単一HTML。
- **デザインコンセプト**: 「英国の不動産屋の店頭ボード（FOR SALE / SOLD）」。ブリティッシュ・レーシンググリーン＋生成りのクリーム、不動産レッドの“SOLD”ラバースタンプ。見出しは英国の老舗看板を思わせるセリフ体（Fraunces）、本文は Manrope。

## 遊び方

1. 画面に**実在の成約物件**が1件表示される（地図・街・通り名・物件種別・所有形態・築古/新築・成約年）。ただし**成約価格は伏せられている**。
2. 地図でどんな立地か、カードでどんな物件かを読み取り、**いくらで売れたか**をスライダーで査定する（£3万〜£250万、対数スケール。±ボタンで微調整）。
3. **査定する**を押すと "SOLD" スタンプとともに実際の成約価格が公開。
4. 成約価格の **半額〜2倍の範囲**に収まれば**査定成功**、外れるとライフ −1。**近いほど高得点**、誤差±10%以内は「ズバリ査定」で大幅ボーナス。
5. 3回ミスで廃業（ゲームオーバー）。連続成功で**査定倍率**が上がり、5件成功ごとに**強化**を選べる。総査定ポイントがスコア。

### スコアと合格ライン

- 合格バンド = 成約価格の **1/2 〜 2倍**（対数誤差 ≤ ln2）。この範囲なら成功、外れるとライフを失う。
- 得点 = `1000 ×（1 − 誤差/ln2）` に連勝倍率を掛けたもの。**±10%以内でズバリ（+600pt）**。
- 連勝が続くほど倍率アップ（×1.0 → 最大×3.0）。ミスで倍率リセット。

### 5件ごとの強化（ローグライト）

5件成功するたびに、3択から1つ選べる。

| 強化 | 効果 |
|------|------|
| 🛡 保険を確保 | 次に査定ミスしても廃業を1回まで免れる（累積可） |
| ❤️ 物件を回復 | ライフを1回復（最大5） |
| 📈 査定倍率アップ | 連勝倍率の上限が上がる（+0.5） |
| 🔍 相場カンペ | 次の5物件、スライダーに「その街の相場帯」を表示 |
| 💷 臨時ボーナス | いますぐ +2,000 pt |

### ランク（総スコアで決定）

冷やかしの内見客 → 見習い査定士 → 駆け出し仲介人 → 一人前の査定士 → 敏腕エージェント → 王立公認 査定士 → **大英不動産卿（30,000pt〜）**

## 物件データについて

**実在の物件・実際の成約価格**を使っている（ヤチタワーが“生成モデル”だったのに対し、こちらは本物）。

- **物件・成約価格**: [HM Land Registry — Price Paid Data](https://landregistry.data.gov.uk/)（英国政府オープンデータ / イングランド＆ウェールズの全不動産取引記録）。鍵不要・CORS対応の公式Linked Data APIから取得。
  - 全国46都市（ロンドン・オックスフォード等の高額エリアから、バーンリー・ミドルズブラ等の安価エリアまで）をランダムに巡り、ページをランダム抽選して毎回違う物件を出す。
  - 競売・非市場取引（`additionalPricePaidTransaction`）と用途不明物件（`other`）、£3万未満/£250万超は除外し、まっとうな一般売買だけを出題。
- **位置（緯度経度・行政区・選挙区）**: [postcodes.io](https://postcodes.io/)（英国ポストコードのオープンAPI、鍵不要・CORS対応）。ポストコードを地図座標に変換して**実際の場所**にピンを立てる。
- **地図**: Leaflet + CARTO Positron タイル（© OpenStreetMap contributors / © CARTO）。

> 立地（地図）と物件スペックの両方が価格の手がかり。ロンドン中心部のフラットと、北部の一戸建ての“相場の逆転”を、地図から読み解くのが査定士の腕の見せどころ。

## 作成の背景

「無料でWebページから使える不動産情報API」を探すのが出発点。当初の第一候補だった国交省の旧・不動産取引価格API（`webland`）は**廃止済み（DNS消滅）**、後継の不動産情報ライブラリ（`reinfolib`）は**要登録キー＋独自ヘッダでCORSブロック**され、ブラウザ直叩き不可だった。日本の公式不動産APIはいずれも鍵登録が必須で、静的サイトのゲームからは使えない。

そこで、**鍵不要・CORS対応で実際の成約価格が取れる**唯一の実用解として英国 HM Land Registry の Price Paid Data に着目。実売価格が地図とセットで取れるため、単なる相場当てを超えて「立地を地図で読む」という遊びが成立した。ハイ＆ロー（[くにくらべ](../kunikurabe/) / [ヤチタワー](../yachitower/)）とは違い、**連続量をスライダーで当てる“Price is Right”型**にすることで、既存ゲームとメカニクスを差別化している。

## 技術構成

- **単一HTMLファイル**（`index.html`）。ライブラリはCDNからバージョン固定＋SRIで読み込み（Leaflet 1.9.4）。単一HTML構成を維持。
- **リアルタイムデータ**: fetch で HM Land Registry / postcodes.io を叩き、先読みキューで次の物件を裏で用意（体感待ち時間を最小化）。取得失敗時は別の街・別ページへ自動リトライ。地図の読み込みに失敗しても物件カードだけでプレイ継続可能なフォールバックを実装。
- **地図**: Leaflet + CARTO タイル。物件位置にカスタムの家型ピンを表示。CDN読み込み失敗時は地図なしで続行。
- **効果音**: Web Audio API でリアルタイム合成（外部音源なし）。iOS の消音スイッチ対策として、初回タップで無音WAVを `<audio>` ループ再生＋`AudioContext.resume()` の完了待ち＋`interrupted` からの自動復帰を実装。
- **メインループ**: `requestAnimationFrame` はタイトル画面から回り、背景の浮遊モチーフ描画と価格カウントアップのトゥイーンを駆動。ループが触る状態は宣言時に全初期化し、try/catch でループが死なないようにしている。
- **スマホ対応**: `viewport-fit=cover` ＋ `env(safe-area-inset-*)`、`touch-action`、ダブルタップズーム無効化（`touchend`/`gesturestart`/`dblclick` を preventDefault）、入力は Pointer Events で統一。ルート要素は `max-width:500px` の縦カラムに制限し、PCワイド画面でも中央寄せ。
- **永続化**: 自己ベストを localStorage に保存（try/catch フォールバック）。

## 実施したテスト

- **構文チェック**: インラインスクリプトを抽出して `node --check`。
- **実ブラウザE2E**（puppeteer-core + ローカルChrome、iPhoneビューポート＋タッチ、ローカルHTTPサーバ経由）:
  - `pageerror` / consoleエラーが**ゼロ**（favicon 404 のみ）。
  - 実データ取得 → 地図タイル＋ピン描画 → スライダー査定 → SOLDリビール（カウントアップ・¥換算・行政区フレーバー）→ 5件成功で強化選択 → 3ミスでゲームオーバー＆ランク表示、の全フローを通過。
  - iPhone SE（375×667）の短い画面でも「査定する」ボタンが常に画面内に収まることを確認。
- **API検証**: HM Land Registry / postcodes.io がいずれも `Access-Control-Allow-Origin: *`・鍵不要で、ページング・成約日フィルタ・成約種別の絞り込みが効くことを事前に確認。全46都市がデータを返すことを検証済み。

### 開発で得た教訓

- 日本の不動産公式API（webland/reinfolib/RESAS/e-Stat）は**廃止済みか鍵必須**で、静的サイトのブラウザから直接は使えない。実売価格が鍵不要・CORS可で取れるのは英国 HM Land Registry がほぼ唯一。
- Leaflet の SRI ハッシュを1文字でも誤ると `integrity` 検証で**JSが丸ごとブロックされ地図が消える**。CDNの実ファイルから `openssl dgst -sha256` で正しいハッシュを算出して固定した。
- 地図上のリボン（`z-index`）がリビールのオーバーレイより上に漏れる不具合を、実ブラウザのスクショで発見・修正。ヘッドレスのロジックテストだけでは見えない見た目の破綻は、実機ビューポートでのスクショ確認が要る。
