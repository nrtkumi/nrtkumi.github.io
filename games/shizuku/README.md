# シズク — 記憶でつくる、深夜のバー

客が頼んだカクテルの**レシピを一瞬だけ記憶**し、棚から**正しい材料だけ**を選んで一杯を仕上げる、記憶系ローグライト。実在カクテルの写真とボトル画像で遊ぶ。

- **プレイ**: https://nrtkumi.github.io/games/shizuku/
- **対応環境**: スマホ縦画面ファースト(タッチ操作)。PC・キーボードでも可。モダンブラウザ。
- **デザインコンセプト**: 深夜のオーセンティック・バー。黒檀と真鍮、琥珀色の一灯。明朝体(Shippori Mincho)のメニュー書き。紫グラデやネオングローは使わない。

## 遊び方

1. 客がカクテルを注文する。写真と一緒に**レシピ(材料)が数秒だけ表示**される。覚えろ。
2. 注文カードをタップすればすぐ「提供」へ進める(残り記憶時間はボーナスになる)。
3. レシピが隠れ、棚にボトルが並ぶ。**レシピにあった材料だけ**をすべて選べ。
4. 全部そろえば提供成功。**速く・コンボを切らさず・記憶時間を残す**ほどチップが伸びる。
5. 間違えると体力(🍒)が減る。客を一定数さばくと夜が明け、**差し入れ(レリック)**で強化できる。
6. 体力が尽きたら閉店。今夜の売上が記録される。

### 差し入れ(レリック)

| 名前 | 効果 |
|---|---|
| 📓 バーテンの手帳 | 覚える時間 ＋1.5秒 |
| 🔮 予感 | 一杯につき間違いを1回まで見逃す |
| 🥃 常連の信頼 | ミスしてもコンボが途切れない(夜1回) |
| ⚖️ 純銀のジガー | チップがすべて ×1.3 |
| 👂 早耳 | 提供開始時、正解の棚を1つ一瞬光らせる |
| 🌸 看板娘 | 体力＋1、最大体力も＋1 |
| ☕ 濃いめの一杯 | スピードボーナスが2倍効く |
| 🎩 上客 | 提供時間 ＋2.5秒 |

## 作成の背景

`/deep-research` による市場調査(2026-06)を踏まえて企画した。

- 中毒性の核は **コンボ・ストリーク・短セッションの磨き込んだループ**にあるという結論(ハイブリッドカジュアルではパズルが2025 Q1収益の48%を占め、磨き込んだコアループのみで成功する `Color Block Jam` が四半期約$25Mを記録)。本作は「覚える→選ぶ」の最小ループにコンボとローグライト強化を重ねた。出典: [gamigion: Top 10 Hybridcasual Games Q1 2025](https://www.gamigion.com/top-10-hybridcasual-games-in-q1-2025-the-great-puzzle-takeover/)
- ローグライトの**ラウンド間強化(メタ進行)**は `Balatro` 以降の潮流。出典: [Stream Hatchet: Indie Roguelike Deck-builders](https://streamhatchet.com/blog/indie-roguelike-deck-builders-on-live-streaming/)
- 既存20作には「記憶(短期記憶リコール)」ジャンルが無く、被らない。

データは **[TheCocktailDB](https://www.thecocktaildb.com/api.php)** の公開API。`random.php` で実在カクテル(写真・材料)を取得し、材料サムネイル(`/images/ingredients/{name}-Small.png`)を棚のボトルに使う。CORS対応・キー不要(テストキー `1`)・無料。

> 注意: TheCocktailDBのテストキー `1` は開発・教育・個人利用向け。アプリストア公開にはPremiumキーが必要(本サイトは個人のホビー用途のため可)。

## 技術構成

- 単一HTML(外部ライブラリなし)。レイアウトはDOM、演出(チップ粒子)とタイマーは `requestAnimationFrame` 1本で駆動。
- カクテルは初回起動時に `random.php` を4並列で取り寄せ、`localStorage` にキャッシュ(次回以降は即プレイ&裏で補充)。通信全滅時は内蔵の古典カクテル10種(テキストのみ)にフォールバックし、無言で壊れない。
- 材料画像は `onerror` で絵文字チップに自動フォールバック。
- iOS消音スイッチ対策: 初回タップで無音WAVを `<audio>` ループ再生 + `AudioContext.resume()` + `onstatechange` 復帰。効果音はWeb Audioでリアルタイム合成。
- スマホ対策: viewport固定、セーフエリア、Pointer Events統一、ダブルタップズーム無効(`touchend` 350ms / `gesturestart` / `dblclick` preventDefault)、`max-width: min(480px,62vh)` の縦カラムでPCワイド画面に対応。
- OGP一式(絶対URL)。`assets/ogp.jpg` は 1200×630。

## テスト

1. `node --check` でインラインJSの構文チェック。
2. puppeteer-core + ローカルChrome + モバイルビューポート(`hasTouch`)で実機相当E2E: タイトル→開店→記憶→提供→コンボ→シフトクリア→レリック選択→難易度上昇までを14ラウンド自動プレイ。`pageerror`/consoleエラーがゼロ(favicon 404を除く)、材料画像404は `onerror` で吸収されることを確認。
3. ローカルHTTPサーバ(`127.0.0.1`)経由でテスト(`file://` だと外部画像でcanvasが汚染される問題を回避)。

## 開発で得た教訓

- 「記憶→選択」型は**選択フェーズでレシピを隠す**だけで、同じ材料タップでも食堂系(満腹食堂)の「速さ勝負」とは全く別の緊張になる。隠す/見せるの一点でジャンルが変わる。
- 材料サムネイルは欠損が普通にある。**最初から `onerror` フォールバック前提**で組むと破綻しない。
