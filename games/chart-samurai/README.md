# チャート侍 (CHART SAMURAI)

正体を伏せた**実在の過去為替チャート**がリプレイ再生されるので、タップで買い→もう一度タップで売り、を繰り返して元手100万円をどこまで増やせるかのタイミングゲーム。
Frankfurter API(欧州中央銀行の日次参考レート)の実データを使い、2000年〜昨年のランダムな9ヶ月間 × ランダムな通貨ペアが出題される。

- **プレイURL**: https://nrtkumi.github.io/games/chart-samurai/
- **対応環境**: スマホ(縦画面・タッチ)/ PC(Space=売買、1・2・3=速度)
- **通信**: 必要(Frankfurter APIをブラウザから直接叩く)

## 遊び方

1. 「出陣」で開始。正体不明のチャート(ペア名・年代は ??? 表示)が右へ伸びていく
2. 画面下の大ボタンをタップで**買い**(資金は全額自動投入)、もう一度タップで**売り**。1ラウンド内の売買回数は無制限
3. 保有中はチャートの該当区間が損益色(緑/赤)に塗られ、含み損益がリアルタイム表示される
4. 約180営業日のリプレイが終わるとラウンド清算。未決済ポジションは自動決済され、「これは 2008年9月〜2009年5月 の USD/JPY でした」とネタばらし
5. 資金を持ち越して全3ラウンド。最終資金がスコア(ベストは端末に保存)

| 要素 | 内容 |
|---|---|
| 出題 | USD/JPY, EUR/USD, GBP/USD, EUR/JPY, GBP/JPY, AUD/JPY, USD/CHF, EUR/GBP から抽選 × 2000年〜昨年のランダム窓(リーマンショックやコロナショックも自然に混ざる) |
| 価格表示 | 開始日=100の指数表示(実レートの値からペアを推測されないため) |
| 倍速 | ×1 / ×2 / ×4(キー 1・2・3) |
| ショート | なし。買い→売りのみ(シンプルさ優先) |
| 称号 | 最終資金に応じて 浪人〜相場の神 の称号が付く |
| シェア | リザルトから X(Twitter)に最終資金と称号をシェア |

## 作成の背景

市場調査の要点:

- 2026年のブラウザゲームはハイパーカジュアルがプレイ時間シェア78.8%を占め、「最初の10秒でルールが分かる」ことがヒットの条件([primepulselogic](https://primepulselogic.com/browser-games-2026-viral-trend/))
- 検索トレンドでも Block Blast・spacebar clicker など「説明不要の即遊べる系」が急伸([Rising Trends](https://www.risingtrends.co/trends/gaming-trends-2026))
- ブラウザゲームシーンはNYT Games系のデイリー/クイズ系とindie HTML5が主流([Dinogame GG](https://dinogame.gg/blog/best-new-browser-games-2026/))

「The Higher Lower Game」(Google検索数の多寡を当てる)以来の**実データ×ゲーム**の系譜を、為替チャートのリプレイ売買に応用した。本物の歴史チャートなので「実はこれリーマンショックでした」というネタばらしの驚きがコア体験。操作はワンボタン(買う/売る)だけでハイパーカジュアルの文法に収まり、サイト内の既存ゲーム(アクション、クイズ、ローグライト等)とジャンルが被らないタイミング/トレード枠。

## 技術構成

- 依存ライブラリなしの単一HTML(チャートはcanvas描画、UIはDOM)
- **使用API(キー不要・無料・CORS対応)**: Frankfurter API v1 — `https://api.frankfurter.dev/v1/{開始日}..{終了日}?base={通貨}&symbols={通貨}` でECBの日次レート(営業日のみ、1999年〜)が1リクエストで取れる。旧ドメイン api.frankfurter.app は301になるため必ず api.frankfurter.dev/v1 を使う
- 出題窓は暦日270日(営業日約185日)を取得して180日に切り詰め。データ不足(休場の多い窓など)は別窓で最大3回リトライ、ネットワーク失敗はエラー画面+再試行ボタン
- リプレイは1倍速で約51秒(3.5営業日/秒)。Yスケールは「見えている範囲」に追従してスムージング、X軸は全期間固定で右端に向かって伸びる
- 価格は開始日=100の指数に正規化して表示(実レート値からのペア推測=後出し知識を防ぐ)
- 売買は `資金 × 価格変化率` の全額自動投入。保有区間はcanvas上で損益色に塗り、建値の点線・含み損益を表示
- 演出: 売買時に刀の斬撃エフェクト+画面フラッシュ(canvas)、損益ポップ、清算時の札束カウントアップ。効果音はWeb Audioでリアルタイム合成(斬撃=ノイズ+下降ソー波など)
- iOS消音スイッチ対策(無音WAVを`<audio>`でループ再生、`AudioContext.resume()`のPromise完了待ち、`interrupted`からの自動復帰)
- 入力はPointer Eventsで統一(`click`不使用)、ダブルタップズーム無効化3点セット、PCワイド画面は縦長カラム(`min(100vw, 520px, 64vh)`)に制限
- canvasのdpr変換は毎フレーム `ctx.setTransform` で再設定(コンテキスト喪失対策)。メインループはタイトル時点から回し、ループが触る状態は全て宣言時に初期化
- ベスト資金・ミュート設定はlocalStorage(try/catchでフォールバック)
- E2E用に `window.CS`(phase / funds / round / holding / speed / roundMeta 等)の読み取り専用フックを公開
- OGP画像はcodex CLI経由のgpt-image-2で生成(浮世絵調・侍のシルエットが金色のチャートを刀で斬る構図+筆文字タイトル、1536x1024 → sipsで1200x800 JPEGに変換)

## 実施したテスト

1. **構文チェック**: インラインscript抽出 → `node --check` パス
2. **ヘッドレスロジックテスト(Node、DOM/fetch/rAFスタブ+合成チャートデータ)**: 全37チェックパス
   - 開始→買い→売り→損益反映→倍速→キーボード操作→自動決済→3ラウンド→リザルト→ベスト保存→タイトル復帰
   - fetch失敗→エラー画面→再試行→復帰
3. **実ブラウザE2E(puppeteer-core + ローカルChrome、390x844・hasTouch、実API、ローカルHTTPサーバ経由)**: 全項目パス
   - 実APIからランダム窓のレートを取得して1ラン(3ラウンド)をタッチ操作のみで完走
   - `pageerror`・consoleエラー 0件、canvasの描画ピクセル確認、スクリーンショットで見た目を目視確認

## 開発で得た教訓

- Frankfurterの期間エンドポイントは**出題チャート1本が1リクエストで丸ごと取れる**うえ日付キーが営業日のみなので、配列化するだけでそのままリプレイ素材になる
- 「実在チャートの正体を伏せる」には期間・ペア名だけでなく**価格の絶対値も隠す**必要がある(107円ならUSD/JPYだと一目でばれる)。開始日=100の指数表示に正規化して解決
- Yスケールを全期間でなく「再生済みの範囲」に追従させると、プレイヤーは未来の値動きの振れ幅をスケールから推測できなくなり、リプレイの緊張感が保てる
